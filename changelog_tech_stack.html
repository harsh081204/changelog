Full tech stack breakdown for the AI changelog writer project

What the user sees and interacts with

Next.js 14 (App Router)
React framework — handles routing, SSR, and API routes in one
Core
App Router
File-based routing. /[owner]/[repo]/page.tsx is the public changelog URL. /dashboard/page.tsx is the private edit view. No config needed.
Server components
The public changelog page fetches directly from Postgres in a server component — no API call needed, no client-side data fetching. Renders as static HTML. Fast.
API routes
The webhook receiver (/api/webhooks/github) and publish endpoint (/api/entries/[id]/publish) live here. No separate Express server needed.
Next.js lets you run the frontend, the webhook API, and the auth layer from one codebase deployed to one place (Vercel). You don't need a separate API server for this project.
Tailwind CSS + shadcn/ui
Styling and component library
UI
Tailwind CSS
Utility-first CSS. You write className="flex items-center gap-3 text-sm" instead of writing separate CSS files. Standard at all modern startups.
shadcn/ui
Pre-built accessible components (Button, Badge, Card, Separator) that you copy into your project and own. Used for the dashboard UI. Not a dependency — you paste the source.
Tiptap (rich text editor)
The draft editing interface
Editor
What it does
The AI writes a draft changelog in markdown. Tiptap renders it as an editable rich text area — like a lightweight Notion editor. Maintainers can tweak wording before publishing.
Why Tiptap and not a plain textarea
Changelog entries have structure — bullet points, bold change types, code names. A plain textarea loses that. Tiptap gives you a real editor with proper HTML output that renders correctly on the public page.
NextAuth.js
Authentication — GitHub OAuth login
Auth
GitHub OAuth provider
Users sign in with their GitHub account. After login you get their GitHub username — which matches the owner field in your repos table. This is how the dashboard knows which repos belong to the logged-in user.
Route protection
The /dashboard route checks the session — only the repo owner can see and edit their drafts. Public /[owner]/[repo] pages are accessible to everyone without login.

The processing engine — runs separately from the frontend

Node.js worker process
Standalone process that runs the job queue — separate from Next.js
Core
Why a separate process
Next.js API routes have a 10-second timeout on Vercel. Fetching a diff + calling an LLM can take 15–30 seconds. The worker runs on Railway with no timeout limit — it processes jobs at its own pace.
TypeScript + tsx
The worker is written in TypeScript and run with tsx — no separate compile step during development. In production it compiles to plain JS first with tsc.
BullMQ
The job queue — sits between the webhook receiver and the worker
Core
What it does
When a webhook fires, the Next.js API route adds a job to BullMQ and immediately returns 200 OK to GitHub. The worker picks it up asynchronously. GitHub doesn't wait for the LLM call to finish.
Retries + deduplication
If the LLM call fails, BullMQ retries automatically with exponential backoff. The jobId is the PR id — so if GitHub re-delivers the same webhook, BullMQ silently ignores the duplicate.
Backed by Redis
BullMQ stores the job queue in Redis. Jobs survive worker restarts — if your Railway worker crashes mid-processing, the job stays in the queue and gets picked up when it restarts.
Octokit
GitHub's official Node.js SDK
GitHub SDK
What it fetches
After a webhook fires, Octokit fetches: the PR title, description, list of changed files, and the actual diffs (patch strings). This is the raw material the LLM classifier works from.
GitHub App auth
Octokit uses a short-lived installation token (not a static PAT) to authenticate. Each token lasts 1 hour and is scoped to one repo installation. More secure than a personal access token.

How and where data is stored

PostgreSQL
The primary database — stores everything
Core
What's in it
Five tables: Installation (GitHub App installs), Repo (tracked repos), ChangelogEntry (one per PR), Change (individual classified changes), and NextAuth session tables.
Why Postgres over MongoDB
Changelog data is highly relational — an Entry has many Changes, a Repo has many Entries, an Installation has many Repos. SQL joins handle this cleanly. MongoDB would require manual reference lookups.
Prisma ORM
TypeScript-first database client
ORM
Type safety end-to-end
Prisma generates TypeScript types from your schema. When you write db.changelogEntry.create({...}), TypeScript knows exactly what fields are required and what types they are. No runtime surprises.
Migrations
Every schema change creates a versioned SQL migration file. Run prisma migrate dev locally and prisma migrate deploy in production. Your schema changes are tracked in git alongside your code.
Prisma Studio
A visual DB browser that comes free with Prisma. Run npx prisma studio — opens a web UI to browse and edit your data. Invaluable during development instead of raw SQL queries.
Redis
In-memory store — used exclusively for the job queue
Queue store
Role in this project
Redis only stores BullMQ job state — pending, active, completed, failed jobs. It does not store application data. Postgres handles all persistent data. Redis is ephemeral and can be flushed without data loss.
Local setup
One line in docker-compose.yml — image: redis:7-alpine. In production, Railway provisions a Redis instance for you automatically when you add the Redis plugin.

The intelligence — how raw diffs become human-readable changelog entries

Anthropic SDK (Claude)
The LLM doing classification and writing
Core AI
What the model does
It receives a PR title, PR description, and a truncated diff. It returns structured JSON — a change type (FEATURE / BUGFIX / BREAKING / INTERNAL), a one-sentence user-facing summary, and a suggested semver bump.
Structured output — not prose
The prompt instructs the model to return only valid JSON. The worker parses this and inserts typed rows into Postgres. If the model misbehaves and returns prose, the try/catch falls back to a safe default.
Context management
Large diffs are truncated to 12,000 characters before being sent. Lock files, minified files, and sourcemaps are filtered out. The diff never exceeds the model's context window and costs stay predictable.
Can swap to OpenAI
The LLM is swappable — same prompt works on GPT-4o
Optional
Why it doesn't matter which one
The classifier is one isolated function. You call it with a string, it returns JSON. Swapping Anthropic for OpenAI is changing 4 lines in one file. The rest of the system doesn't care.

Where everything runs

Vercel
Hosts the Next.js frontend + API routes
Frontend host
Zero config deployment
Push to main branch → Vercel detects Next.js → builds and deploys automatically. Your app is live at changelog-writer.vercel.app within 2 minutes. No server config, no Nginx, no SSL setup.
Free tier is enough
Vercel's free hobby plan handles this project's traffic easily. The only limit is that serverless functions timeout at 10 seconds — which is why the LLM processing happens in the Railway worker, not here.
Railway
Hosts the Node.js worker + Redis + Postgres in production
Backend host
No timeout limits
Unlike Vercel, Railway runs your process continuously. The BullMQ worker sits there permanently, picking up jobs as they arrive. An LLM call that takes 30 seconds is fine here.
One-click Redis and Postgres
In the Railway dashboard, click "Add plugin" → Redis or Postgres. Railway provisions it and injects REDIS_URL and DATABASE_URL automatically into your worker's environment. No separate database service needed.
Docker deployment
The worker has a simple Dockerfile. Railway detects it, builds the image, and deploys. Costs ~$5/month on the starter plan for a project of this scale.
Docker (local dev only)
Runs Postgres and Redis locally without installing them
Dev tool
One command local setup
docker-compose up -d starts both Postgres on port 5432 and Redis on port 6379. No local installation of either service. Stop with docker-compose down. Data persists in a Docker volume between restarts.
Resend
Sends the post-publish email summary (optional feature)
Optional
What it sends
When a maintainer publishes a changelog entry, Resend fires an email to anyone who subscribed to that repo's changelog. Clean HTML email with the entry content. Free tier: 3,000 emails/month.

The GitHub integration — what connects your app to repos

GitHub App
The identity your app uses to interact with GitHub
Core
App vs OAuth vs PAT
Personal Access Token
Tied to one user. If that person leaves, the token breaks. No webhook support.
OAuth App
User authorises your app. You get their token. Still user-scoped, not repo-scoped.
GitHub App ✓ (what we use)
Installs on repos directly. Scoped permissions. Short-lived tokens. Receives webhooks automatically. The right choice for any production tool.
A GitHub App is what Vercel, Linear, and Dependabot use to integrate with your repos. It's the professional-grade approach and the one that teaches you the most.
GitHub Webhooks
How GitHub tells your app something happened
Event system
Events you subscribe to
Two events: pull_request (fires on PR merge) and push (fires on direct commits to main). GitHub sends an HTTPS POST to your webhook URL within seconds of the event.
HMAC signature verification
Every webhook has a X-Hub-Signature-256 header — an HMAC-SHA256 signature of the payload using your webhook secret. You verify this before processing anything. Rejects spoofed requests.
Octokit SDK
Official GitHub TypeScript SDK
SDK
Packages used
@octokit/app handles GitHub App auth (JWT signing, installation token generation). @octokit/rest provides typed methods for every GitHub API endpoint — pulls.get(), pulls.listFiles(), etc.

Why these choices were made — not just what they are

Why TypeScript everywhere
Not just for code quality
Prisma generates types from your schema. Octokit ships types for every GitHub API response. The LLM classifier returns a typed interface. When you pass data from one layer to another, TypeScript catches mismatches at compile time — not in production when a missing field crashes a job.
Why Next.js handles webhooks
Instead of a separate Express server
The webhook receiver does almost nothing — verify signature, add job to queue, return 200. That's 15 lines of code. Running a separate Express server for 15 lines is unnecessary overhead. A Next.js API route is perfect for this. The heavy work happens in the Railway worker.
Why BullMQ and not just a setTimeout
The difference between a toy and a real system
What breaks without a queue
If you call the LLM inline in the webhook handler: GitHub waits 30 seconds and retries, causing duplicate processing. A server restart loses in-flight jobs. 10 simultaneous PRs overwhelm the API rate limit. BullMQ solves all three — jobs persist in Redis, retries are automatic, concurrency is configurable.
The full data flow in one sentence
From commit to published changelog
GitHub fires a webhook → Next.js API route verifies signature + enqueues a BullMQ job → Railway worker dequeues it → Octokit fetches the PR diff → Claude classifies the changes into typed JSON → Prisma writes a DRAFT entry to Postgres → maintainer logs in via GitHub OAuth, edits the Tiptap draft, clicks publish → entry status flips to PUBLISHED → public Next.js page reads it from Postgres and renders it as HTML.
